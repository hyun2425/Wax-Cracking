# 개발 요구사항

이 문서는 온라인 왁스볼 크래킹 프로젝트의 기능 요구사항과 개발 기준을 정리합니다. 인턴이나 새 작업자가 프로젝트 의도를 빠르게 이해하는 것을 목표로 합니다.

## 1. 프로젝트 목표

왁스볼을 선택하고, 냉동 시간에 따라 단단함이 달라진 왁스볼을 클릭해 깨는 ASMR 시뮬레이션을 만든다.

실제 왁스볼은 한 번 깨면 다시 사용할 수 없지만, 웹에서는 여러 종류의 왁스볼을 반복해서 체험할 수 있도록 한다.

## 2. 기술 스택

### 프론트엔드

- Next.js
- TypeScript
- TailwindCSS
- Nanum Gothic 폰트
- Vercel 배포

### 백엔드

- Spring Boot
- Java 21
- Gradle
- Render 배포

### 데이터베이스

- 현재 사용하지 않음
- DB 연결 설정, 엔티티, 마이그레이션 도구는 아직 추가하지 않음

## 3. 주요 사용자 흐름

1. 사용자가 왁스볼 종류를 선택한다.
2. 사용자가 왁스볼을 냉동실에 넣는다.
3. 냉동 시간이 지나면서 왁스볼의 단단함이 변한다.
4. 사용자가 마우스 클릭으로 왁스볼을 누른다.
5. 클릭 횟수나 강도에 따라 균열이 생기고 확장된다.
6. 왁스볼이 완전히 깨지면 ASMR 사운드가 재생된다.

## 4. 왁스볼 종류

### 두바이 왁스볼

- 초콜릿빛 외피
- 금빛 라인 장식
- 묵직한 파열음
- 고급스럽고 단단한 이미지

### 솜사탕 왁스볼

- 분홍, 하늘, 보라색 조합
- 부드럽고 가벼운 이미지
- 가벼운 바스락 소리

### 청사과 왁스볼

- 연두색 표면
- 산뜻하고 시원한 이미지
- 깨끗한 크랙 사운드

## 5. 냉동 시간 규칙

사용자가 오래 기다리지 않도록 게임 안에서는 `1초 = 1분`처럼 처리하는 것을 목표로 한다.

| 냉동 시간 | 상태 | 사운드 예시 |
| --- | --- | --- |
| 0~5분 | 말랑 | 톡 |
| 5~10분 | 보통 | 빠각 |
| 10~15분 | 단단 | 바삭 |
| 15~20분 | 매우 단단 | 쩌억 |
| 20분 | 최고 경도 | 크그작 |

## 6. 파괴 진행 규칙

왁스볼은 클릭에 따라 단계적으로 깨진다.

1. 클릭 시작
2. 작은 균열 발생
3. 균열 확장
4. 완전 파괴
5. 파괴음 재생

냉동 시간이 길수록 균열 효과와 파괴음이 더 선명하게 느껴지도록 한다.

## 7. 프론트엔드 요구사항

- 첫 화면에서 프로젝트 주제가 바로 보여야 한다.
- 왁스볼 종류, 냉동 시스템, ASMR 사운드, 파괴 흐름을 한눈에 볼 수 있어야 한다.
- 한국어 문구를 기준으로 작성한다.
- 모바일에서도 텍스트가 겹치거나 잘리지 않아야 한다.
- 폰트는 Nanum Gothic을 사용한다.
- 백엔드 연결 상태는 `/api/health` 응답으로 확인한다.

## 8. 백엔드 요구사항

- 기본 상태 확인 API를 제공한다.

```text
GET /api/health
```

정상 응답 예시:

```json
{
  "status": "ok",
  "service": "wax-cracking-backend",
  "database": "disabled"
}
```

- 현재는 DB를 사용하지 않는다.
- 프론트엔드 배포 도메인에서 API를 호출할 수 있도록 CORS를 설정한다.

## 9. 배포 요구사항

### Vercel

- 프론트엔드 Root Directory는 `apps/frontend`로 설정한다.
- 백엔드 주소는 `NEXT_PUBLIC_API_BASE_URL` 환경 변수에 넣는다.

### Render

- 백엔드는 `render.yaml`을 통해 배포한다.
- Blueprint Path는 `render.yaml`이다.
- Health Check Path는 `/api/health`이다.

## 10. 작업 규칙

- 작업 전 `git pull --ff-only origin main`으로 최신 상태를 맞춘다.
- 프론트엔드 작업은 `apps/frontend` 안에서 진행한다.
- 백엔드 작업은 `apps/backend` 안에서 진행한다.
- 작업 후 프론트엔드는 `npm run lint`, `npm run build`를 확인한다.
- 백엔드 변경 후에는 `.\gradlew.bat test`를 확인한다.
- 의미 없는 루트 파일을 새로 만들지 않는다.
